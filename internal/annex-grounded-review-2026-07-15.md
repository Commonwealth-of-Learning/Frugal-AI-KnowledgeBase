# Annex-grounded review — 2026-07-15

A review of the knowledge base with `docs/reference/sovereign-education-ai-reference-architecture.md` (the adaptation of COL's Technical Annex) as the grounding document, plus a proposal set for updating the annex itself for the agentic era (agents, harnesses, loop engineering). Method: seven parallel mapping passes (annex→KB coverage matrix, concepts/landing, guides, components/operations, agentic-content inventory, internal decision constraints, reference-page consistency), two synthesis analysts, and an adversarial verification pass on every proposal (each verifier instructed to refute: evidence checked against the tree, synonym-searched for existing coverage, cross-checked against CLAUDE.md decisions, the gap reviews, and `internal/annex-v2-proposed-changes.md`). Of 32 proposals, 12 were confirmed as written, 15 confirmed in substance with corrected details (recorded below in corrected form), 2 dropped, and 3 carry a verification caveat noted inline. Line numbers refer to the working tree at commit 340ee14 (with the teacher-hexagon section removed from the reference page on user instruction, 2026-07-14).

**Headline findings**

1. The annex-v2 pending change list is entirely non-agentic. All fourteen v2 changes (`internal/annex-v2-proposed-changes.md`) are measurement rigour — task–model fit, M&E measurement sheet, privacy-airlock test plan, open-source scan, freshness register. The annex (v1 and v2 draft alike) contains **zero agentic vocabulary**: no "agent", "harness", "MCP", "skill", or agent-memory language; "loop" appears only inside "Teacher-in-the-Loop"; "autonomy" only as "strategic autonomy" (§4.2). The KB is ahead of its own policy companion — the three-governance-surfaces model, per-action approval, loop ownership, written skills, and memory governance are all documented publicly with no annex counterpart. Part 3 turns that into a 12-item proposal set for COL.
2. Coverage against the annex is broad but thin at the edges: of 19 annex elements, one is covered (Layer F, the model layer), sixteen are partial, and two are missing entirely (§3.1 interoperability behaviours; §3.2/§7.1 DPI). Most partials are already-known further work (runbooks, pilot guidance, plan-008); Part 2 lists only the gaps that are *not* already tracked.
3. Three real consistency defects, one of them on the grounding document itself: a stale indicator citation on the gateway layer and glossary, an unscoped "one home" sentence on the stack page, and dislocated §7/§8 table content in the annex adaptation (present since the rebuild commit — a conversion artefact needing a check against the source document).

## Part 3 — Reference-architecture update proposals for the agentic era

**Routing rule (applies to every item):** the public reference page is a faithful adaptation of COL's authoritative annex; per the recorded convention (`internal/annex-v2-proposed-changes.md:3`), proposed annex changes are collected for COL review, **never applied to the page**. These twelve proposals are KB-originated input to the annex's next revision (v3), kept separate from the v2 file (which extracts COL's own highlighted draft). None duplicates a pending v2 change — the v2 set is entirely non-agentic (verified item by item). The still-open owner line on the annex feedback channel (gap-review-2026-06-12 C3) applies to this collection too. All proposals are worded for the annex's options-menu register and preserve the v2 guardrails (teacher approval unchanged or strengthened; cloud burst non-default; no high-stakes automated decisions).

Every KB citation below was verified verbatim by the adversarial pass. Interdependencies: A1 defines the terms the others use; A2+A3+A5 form the control set that A8 and A9 then audit.

**A1 · Agents belong in the layered model (high; §3 Layer C row, scope note, §10 glossary).** The annex contains no occurrence of "agent"; Layer C lists only single-output drafting tasks. Propose: (1) amend Layer C to name agents as a subtype of Application Services — applications that take multi-step actions (plan, call tools, edit files, run commands) — with outputs governed by Layer B / §5 risk-tiered oversight, and an explicit note that an agent's tool or MCP traffic does not pass the Layer E privacy airlock and requires its own allowlist control; (2) clarify the scope note so agentic task execution under review gates is within the baseline, while high-stakes automated decision-making stays out of scope regardless of whether an agent performs it; (3) add agent, agent loop (harness), tool egress, written skill, and MCP to the annex's own glossary (the pending v2 glossary additions are Quantisation, QAT, and Task–Model Fit only). KB grounding: application-layer.md:14-16; glossary.md:16-18. Optional KB-side companion (legitimately public, since the crosswalk table is KB-authored commentary): extend the "Application Services | Application layer" crosswalk row to note agents are included there as an Application-layer subtype.

**A2 · The second egress surface (high; §5.3:245, §4.4, §3 Layer C row).** The annex asserts "Regardless of tier, all processing passes through the privacy airlock" — falsified for agents, whose tools and MCP servers make their own network calls. Propose: qualify §5.3 (model-bound processing passes the airlock; tool/MCP traffic is a second external-processing path that does not); extend §4.4's envelope discipline to tool egress — allowlist of permitted tools, MCP servers, and destinations; no network access by default; each network-reaching tool reviewed as a component before adoption; payload rules extended to tool traffic. Surface the control in the §3 layer table under **Layer C** core components, not Layer E (the KB crosswalks Layer E to the gateway, which governs model egress only; tool egress is governed at the application layer, which crosswalks to Layer C). KB grounding: application-layer.md:28; coding-agent.md:142; gateway-layer.md:38 — the three-governance-surfaces model is the ready-made framing.

**A3 · Risk-tier oversight extended from outputs to actions (high; §5.1 + quick test).** The annex's tiers classify outputs only; agents also act. Propose an action-oversight rule orthogonal to the tiers: where an application acts (edits files, runs commands, calls side-effecting tools), oversight applies per action as well as per output — a plan-only review mode before execution, explicit approval for each side-effecting action, a scoped working directory, denial outside scope. Extend the classification quick test: "Does the task involve an agent taking actions with side effects? → per-action approval applies in addition to the output tier." Note the approval step assumes AI literacy (reviewers must be able to see and understand what an agent did). KB grounding: coding-agent.md:93-97,:125,:128-134; manim-animator.md:109; orchestration-layer.md:61; README.md:76.

**A4 · Sovereign operation extends to the loop (high; §2.1:83, §3.1, App A).** The sovereignty principle covers data and audit logs only. Propose: where agents are used, the agent loop (harness) — the orchestration that assembles context, calls tools, and keeps memory between steps — remains under national or institutional control, because whoever operates it sees everything the agent sees while it works, and residency rules alone do not capture working knowledge experienced by an externally run loop. Frame the stake as delegation capacity: whether an institution has agents working for it, or only works inside someone else's. Place the procurement questions in §3.1's procurement-relevant behaviours and as Appendix A self-check items (the v2 draft's own pattern), not in §1: who operates the harness; where its logs and memory accrue, under what retention and review; whether the model behind it can be substituted without surrendering the loop. KB grounding: application-layer.md:18-20; orchestration-layer.md:39; README.md:34. (Internal grounding — three-generations table, `internal/frugal-ai-principles.md:18-26` — informs the argument but must never be cited publicly.)

**A5 · Agent memory as a named retention surface (medium; §5.3:249, §2.2:95).** The retention controls name prompts, outputs, and logs only. Propose: for any agentic deployment, define what an agent is permitted to remember across sessions, where that memory is stored, how long it is retained, and who can review it; an agent's working context and memory fall under the same residency and access expectations as stored records, since memory accumulates institutional knowledge even when nothing is stored in the traditional sense. KB grounding: orchestration-layer.md:49; glossary.md:24-26.

**A6 · Written skills as a third versioned artefact class (medium; §2.2:97, Layer D/H rows, §5.1:231).** The annex versions two artefact classes (knowledge base, model configurations). Propose adding written skills and agent instruction files as a third: short written procedures an agent loads into context direct its behaviour, so they are reviewed and version-controlled like any other configuration; register them in Layer H's registries; connect to §5.1's Tier 2 feedback loop — corrections identified through review are folded back into the skill, giving the annex a concrete artefact for "errors identified through audit feed back into system improvement". KB grounding: coding-agent.md:105,:115; manim-animator.md:74; application-layer.md:34; glossary.md:112-114.

**A7 · Staged Tier 1 review as an optional pattern (medium; §5.1:229, §6.3).** The annex's Tier 1 gate is a single pre-release approval; §6.3 names review capacity as the scale constraint but offers only tiering against it. Propose describing staged Tier 1 review as an **optional** pattern for generated artefacts (animations, complex documents, multi-part materials): a plain-language plan or storyboard approved before production begins, plus an explicit inspection checklist at release. Staging moves teacher judgement to the cheapest point without weakening approval-before-learner-release — it strengthens it with an earlier review point (compatible with the v2 guardrail). KB grounding: manim-animator.md:78,:88-94,:101,:111.

**A8 · Layer H components and §8 indicators for agentic use (medium; Layer H:115, §8 table).** *(Substance verified against the annex, the v2 pending set, and the KB inventory; this item and A9 did not get an independent adversarial pass — their verifiers were the two that could not be re-run.)* Propose an "Agent oversight" domain for the §8 indicator table: completeness of agent action logs (every file edit and command recorded with its approval decision); tool-call audit coverage and allowlist conformance (share of tool calls logged; no tool egress outside the allowlist); per-action approval and rejection rates; a defined review cadence for agent memory. Extend Layer H's core components with agent action logs and a registry of approved tools and MCP servers, parallel to the model and knowledge-base registries. Extends, does not duplicate, the pending v2 Appendix E (nine metrics, none agentic); reconcile if COL accepts v2 first. KB grounding: coding-agent.md:128-134; orchestration-layer.md:47-51; gateway-layer.md:51.

**A9 · Appendix A subsection for agentic use (medium; App A).** *(Same verification caveat as A8.)* Propose "Agentic use (if agents are deployed)" in the checklist's existing style: each of the three governance surfaces (local actions, model egress, tool egress) has a named control; tools and MCP servers allowlisted with no network access by default; side-effecting actions require per-action approval with a plan-only review mode available; agent memory retention documented (what, where, how long, who reviews); written skills versioned and reviewed; agent action logs tamper-evident; agents remain single-operator tools until an operational runbook and review workflow exist for shared use. KB grounding: gateway-layer.md:38,:51; application-layer.md:22-30; pilot.md:12.

**A10 · Governed coding agents as a §6.1 capacity mitigation (medium; §6.1:267-269).** The annex names the MLOps capability barrier; its mitigations are institutional only. Propose a technical mitigation: governed coding agents as a force-multiplier for the small teams operating the sovereign stack — drafting configurations, scaffolding components, maintaining documentation under human review. State the required controls directly (not by reference to sibling proposals): review-first per-action approval of edits and commands; model calls routed through the gateway inside the envelope; tools allowlisted with no network access by default, since tool egress does not pass the gateway. Second-order benefit: a team that operates a readable open-source agent also builds the capacity to govern agents generally. KB grounding: coding-agent.md:26,:39-42 (OpenCode's dual role).

**A11 · Agentic workloads strengthen the local-default case (medium; §4.2:158, §4.4).** The annex's cost framing is pre-agentic ("cost volatility", "cost predictability"). Propose a §4.2 paragraph: an agent loop consumes many times the tokens of a single reply (retries, tool calls, multi-step work), so per-token cloud pricing turns autonomy into a rationed resource in exactly the contexts this annex serves, while local inference makes the marginal cost of a long-running agent task close to zero once the modest hardware §1 already assumes exists. Corollary in §4.4 envelope vocabulary: burst applies per hard step inside a locally run loop, never to the loop itself, so only the minimised per-step payload crosses the envelope while the run record and the agent's working state fall under what-must-remain-in-country. KB demonstration: manim-animator.md:109 (loop and actions gated locally; model calls including controlled burst gateway-governed). Internal grounding (`internal/frugal-ai-principles.md:58-60`) never cited publicly.

**A12 · Agent harnesses as a sixth §4.3 open-source category (low; §4.3:176-181).** Propose: agent harnesses and coding agents — open-source agent loops that connect to locally served model endpoints and gate file and command actions behind per-action review. Selection criteria the KB documents: a codebase readable enough for the local team to study the loop it will govern; support for local OpenAI-compatible endpoints; review-first per-action permission gating — exercised in practice by the documented OpenCode path, verified (though not documented as paths) for Tau and Pi. If adopted, the pending v2 Appendix G scan needs a matching dated subsection (the v2 §4.3 note routes every category to Appendix G).
