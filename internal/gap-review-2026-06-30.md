# Gap review — rollout consistency, content quality, and the audit tool

Date: 2026-06-30. Base head: `2606480 Make Gemma 4 12B the default model across all guides`. Scope: all of `docs/` (36 pages), `docs/SUMMARY.md`, internal maintainer machinery (`internal/handoff-2026-06-12.md`, `internal/editorial-guide.md`, `internal/editor-workflow.md`, `internal/naming-registry.md`, `internal/templates/`), and `internal/tools/editorial_audit.py` itself. Method: a five-dimension review (rollout consistency, content quality, internal-doc staleness, structure/tooling hygiene, audit-script code review) with every candidate finding independently re-verified by a second read of the cited file before being accepted; 30 of 30 candidates confirmed. Findings only; no public pages were edited.

Baseline: the 2026-06-12 gap review's same-day fixes (commit `247e4f2`) and handoff refresh (`ac6e5b0`) landed cleanly. Its open items — B1 (administrative-agent guide), B2 (Dify/serving-engine runbooks), B3 (pilot serving guide), B4 (SGLang/llama.cpp cards), B5 (llms.txt confirmation), B6 (teacher-only tier wording), C1 (confidence-label audit heuristic), C3 (annex v2 owner line) — are all still open exactly as described there; this review does not re-litigate them. Two commits landed after that review: `0c370ec` ("Add model comparison and hardware routing") and `2606480` ("Make Gemma 4 12B the default model across all guides"), which switched the documented first-path default model from Qwen3.5-9B to Gemma 4 12B across most of `docs/` and `internal/naming-registry.md`. The gaps below are what that rollout missed, plus new content-consistency and audit-tool defects this review's closer read surfaced independently of it.

**A3. `docs/SUMMARY.md:38-40`** still lists `Model: Qwen3.5-9B` and `Model: Qwen3.6-35B-A3B` before `Model: Gemma 4 12B` in the sidebar. **A5. ## D. `internal/tools/editorial_audit.py` — code-quality findings

The mechanical audit currently passes clean from both the repo root and `docs/`, and none of the below currently produces a live false positive — but each is a reproduced, real defect rather than a theoretical one.

**D1 (high). Absolute-link mis-resolution.** `check_internal_links` (line 431, `(path.parent / cleaned).resolve()`) and `discover_linked_docs` (line 393, `(DOCS_DIR / target_path).resolve()`) both rely on `pathlib`'s `/` operator, which silently discards the left operand when the right side is an absolute (leading-`/`) path. A link written GitBook's standard root-relative way, e.g. `[Math tutor](/getting-started/math-tutor.md)`, resolves against the filesystem root instead of `docs/` and is reported as broken even when the target exists. No current page uses this link style, so the bug is latent. Fix: strip a leading `/` and join explicitly against `DOCS_DIR` before resolving.

**D2 (high). Code-fence blind spot.** Only `check_internal_links` skips fenced ` ``` ` blocks. `scan_patterns`, `check_public_site_names`, `scan_public_placeholders`, and `check_first_screen_abbreviations` all scan raw lines, including inside Mermaid diagrams (already used in four public pages with `Label[Node text]` syntax). A node label like `[Configuration value]` would trigger a false placeholder warning, since "value" is one of `PLACEHOLDER_CUE_WORDS`; reproduced on a synthetic doc. Fix: factor `check_internal_links`'s `in_code_block` toggle into a shared helper and reuse it across all five scanners.

**D3 (medium). Case-sensitive style regexes.** `\bMac Mini\b`, `\b24(?:Gb|GB)\b`, and `\bFrugal AI Knowledge Base\b` (site-name check) have no `re.IGNORECASE`, unlike every other pattern in the same lists. Reproduced: "MAC MINI", "24gb", and "FRUGAL AI KNOWLEDGE BASE" all pass silently. Fix: add `re.IGNORECASE`, and widen `24(?:Gb|GB)` to allow a space.

**D4 (medium). Whitespace-fragile required-entry check.** `check_required_summary_entries` compares `SUMMARY.md` lines to the required list by exact string equality after only outer `.strip()`. Reproduced: changing `* [Math tutor](...)` to `*  [Math tutor](...)` (two spaces, renders identically) causes a false "required summary entry missing" warning. Fix: normalize internal whitespace before comparing, or parse with the same `link_pattern` regex `check_summary_structure` already uses and compare `(label, target)` tuples.

**D5 (medium). Anchor fragments are never validated.** `clean_target` strips `#fragment` before either link check runs `path.exists()`; neither parses the target file's actual headings. A stale or mistyped anchor on an otherwise-valid link, e.g. `[Math tutor](math-tutor.md#wrong-heading)`, produces no warning. CLAUDE.md notes the team currently avoids anchors by convention — this is the missing tooling backstop if that convention lapses. Fix: when a fragment is present, slugify the target's headings and check membership.

**D6 (medium). Port-binding regex misses common forms.** `binding = re.compile(r"-p\s+(?:\d{1,3}(?:\.\d{1,3}){3}:)?(\d{2,5}):\d{2,5}")` only matches Docker's short `-p` flag; it doesn't match `--publish` or a docker-compose YAML `ports:` list item. Every current guide uses the plain `-p` form, so this is latent — but the still-open Dify and serving-engine runbooks (B2 in the 2026-06-12 review) will plausibly need compose syntax. Fix: add a `--publish` alternation and a second pattern for compose-style list items.

**D7 (medium). Over-broad "unverifiable filler" pattern.** `\b(?:not found|cannot be found|could not be found|not available)\b` matches the bare phrase anywhere in prose, not just placeholder leftovers. Reproduced: "If GPU offload is not available, the runtime falls back to CPU" triggers the warning despite being legitimate technical documentation. No current page hits this (existing fallback prose happens to say "unavailable," one word), but the still-open llama.cpp/SGLang cards and serving-engine runbooks will plausibly need this kind of fallback language. Fix: narrow the pattern to the placeholder-leftover shape (e.g. anchor to bracketed/citation-style filler) rather than matching the bare phrase anywhere.

**D8 (low). Dead state.** `LinkedDoc.summary_line` and `.target` are populated by `discover_linked_docs` but never read anywhere else in the script. Fix: drop the fields, or use `summary_line` to report the originating `SUMMARY.md` line on downstream warnings for linked pages.

**D9 (low). Unscoped marker checks.** `check_layer_tag` and `check_guide_hint` search the whole document for `_Layer:`/`Expected time:`/`Level:` rather than the top-of-page block their own docstrings describe, unlike `first_screen_lines` (already used by `check_first_screen_abbreviations`), which scopes correctly. A hint block that moved or was duplicated elsewhere in the body would pass silently. Fix: reuse `first_screen_lines` or an equivalent fixed window after the H1.

## Suggested order

1. **Same-day, no judgment calls:** A1–A5 (rollout consistency — pick one model-size figure and propagate it), C3 (template label sync), C6 (LM Studio port mention).
2. **Same-day, small wording decisions:** B1, B3, B5, B7 (clear factual/taxonomy corrections); C5 (banner text).
3. **One careful pass:** D1 and D2 (the two high-severity audit-script bugs) — both are correctness bugs in the one piece of real code in the repo, worth fixing together since both touch the same line-scanning pattern.
4. **Needs a maintainer call:** B2 (does LiteLLM's "Candidate" wording change?), B4 (validate-then-document, or caveat-then-keep, the DGX Spark recipe?), B6 (confirm Tier 1 is the right classification for the animator before adding it), C1/C2 (backfill `offline-chat-service.md` or repoint the exemplar — this also changes what "the" Guide pattern is, so it's a small decision, not just a typo fix).
5. **Hardening, lower urgency:** D3–D9 (the audit script's remaining latent gaps — none currently misfires, but several map directly onto content the still-open B2/B4 runbooks from the 2026-06-12 review will introduce).

The pattern across A1–A5 and B1–B7 is the same one the 2026-06-12 review named for `internal/` vs `docs/`: the mechanical audit verifies structure (links, frontmatter, required entries, role prefixes) but has no way to check that a *fact restated in three places* — a model-size range, a taxonomy label, a "built vs. later increment" claim — still agrees with itself after one of those places changes. The Gemma 4 12B rollout (commits `0c370ec`/`2606480`) is a clean example: it correctly updated `mac-mini-24gb.md` and missed two sibling pages stating the identical fact. None of D1–D9 would have caught A1–A5 even after being fixed, since cross-page factual consistency isn't a regex-checkable property; the dependable fix is procedural — when a worked example or default changes, grep for every other place the same fact is restated before considering the change complete, the way `internal/naming-registry.md`'s own "update the examples, not the principles" principle already prescribes.
