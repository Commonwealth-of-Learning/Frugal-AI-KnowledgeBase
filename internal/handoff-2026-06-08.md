## Editorial Rules

Public pages should:

- avoid direct second person (`you`, `your`, `yours`);
- use British/Commonwealth spelling;
- keep introductions short and grounded;
- label estimates and source-listed values clearly;
- state development, pilot, and production boundaries;
- name human review for learner-facing use;
- keep commands short and include verification steps;
- avoid long shell scripts, heredocs, and customised code blocks unless necessary;
- remove empty sections instead of saying information is unavailable or cannot be found.

The current landing page was intentionally reverted to a simpler builder-first version in commit `475a334`. Do not reintroduce the partner/ministry-first landing rewrite unless requested.

## Audit And Verification

Run these before committing public documentation changes:

```bash
python3 internal/tools/editorial_audit.py
```

Also run the audit from inside `docs/` to catch path assumptions:

```bash
python3 ../internal/tools/editorial_audit.py
```

Run whitespace checks:

```bash
git diff --check
```

For staged changes:

```bash
git diff --cached --check
```

Useful focused public wording scan:

```bash
rg -n "\byou\b|\byour\b|\byours\b|offline chat service|Offline chat service|Open WebUI operations|Mac Mini|24Gb|24GB|not found|cannot be found|could not be found|not available|TODO|TBD" docs/README.md docs/SUMMARY.md docs/getting-started docs/operations docs/components docs/concepts docs/guides docs/reference
```

The audit follows links from `docs/SUMMARY.md`. It does not fully validate unlinked legacy files unless they are linked.

## Git And Ignore Gotchas

`.gitignore` currently contains:

```text
reference/
```

This ignore rule hides `docs/reference/` from normal discovery and staging even though some reference files are tracked.

Consequences:

- `rg --files` may not show `docs/reference/*`.
- Normal `git add docs/reference/file.md` may fail.
- Use `git ls-files` to see tracked files.
- Use `git add -f docs/reference/file.md` when changing tracked or new reference pages under `docs/reference/`.

This should probably be cleaned up in a future housekeeping pass. If the intent is to ignore only a root-level working library called `reference/`, change the rule to `/reference/` and verify before committing.

There are tracked `.DS_Store` files:

- `.DS_Store`
- `internal/.DS_Store`

Leave them alone unless the user asks for repository cleanup.

## How To Add A New Public Page

1. Confirm the page has a reader job and belongs in the public path.
2. Check `internal/naming-registry.md`.
3. Use the closest template from `internal/templates/`, but remove unused sections.
4. Draft under `docs/`.
5. Link it from `docs/SUMMARY.md` only when it is ready for readers.
6. Run the audit and wording checks.
7. If the file is under `docs/reference/`, stage it with `git add -f`.
