## What this is

GitBook documentation for the Commonwealth of Learning (COL) Frugal AI initiative. Public content lives under `docs/` with navigation in `docs/SUMMARY.md`; internal maintainer material lives under `internal/` and is never linked from public navigation. The knowledge base is developer and technical first, with stakeholder framing on dedicated pages.

## The model (the spine)

A Frugal AI system is a stack of substitutable, optional layers, governed at one boundary:

- **Infrastructure → Inference → Orchestration → Application**, with the **Gateway as the sovereignty envelope**: a real layer in the request path, the single home for governance, and the operational form of the reference architecture's privacy airlock and cloud-burst controls.
- An **agent is an Application subtype**, not a layer.
- The **frugal floor** (Infrastructure + Inference + Application, gateway local-only, no orchestration) is the local chat build.
- Agents have **three governance surfaces**: local actions (review, scoped permissions), model egress (the gateway), and tool/MCP egress, which does **not** pass the gateway and is governed at the application layer.

The first build is the Local AI chat service: Mac mini 24 GB → Ollama → Qwen3.5-9B → Open WebUI. A maths through-line runs across the layers: local chat → math tutor (SymPy exact compute, orchestration) → AI gateway (controlled cloud burst, redacted) → Manim animator (advanced coding-agent build).

Sections, in sidebar order: Start here (quickstart, chat service guide) · Concepts (the stack, example applications) · Build further (math tutor, curriculum advisor, AI gateway) · Advanced (coding agent, Manim animator) · one section per layer, each opening with its layer overview and nesting its component cards · Environments (development, pilot, production) · Operations (overview, chat-service runbook) · Reference and policy (glossary, reference architecture, use with an AI agent).

- **OpenCode has a dual role**: an example agent and the builder's tool a local team uses to extend and maintain the stack (the capacity-building story).
- **Port policy**: keep upstream default ports, register every host port in the naming registry's Port Allocations table, and link to official docs for non-default configuration.
- **Operations policy**: every operable component appears in `docs/operations/operations-overview.md`, covered by a runbook or marked further work.
